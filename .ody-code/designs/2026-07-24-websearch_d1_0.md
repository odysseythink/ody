# WebSearch 重构 D1.0 详细设计

## Scope In / Scope Out

### Scope In [C:USER]
- 完全删除 Rust 端现有 web search 运行时路径：hosted `web_search`、standalone `web/run` 扩展、`ody-api::SearchClient` / `alpha/search` 端点、provider/model 层 `web_search` 与 `supports_search_tool` 能力门控、`web_search_mode` 配置及 `Feature::StandaloneWebSearch` / `Feature::WebSearchCached` / `Feature::WebSearchRequest`。
- 新建 `ody-web-search` crate，对齐 TS `ody-code`：
  - `WebSearchProvider` trait / `WebSearchResult` / `WebSearchOptions`；
  - `WebSearchProviderRegistry` + `FallbackWebSearchProvider`；
  - 12 个 provider 实现：DuckDuckGo、Bing、SerpApi、SearchApi、Serper、Baidu、Serply、SearXNG、Tavily、Exa、Perplexity、Moonshot。
- 新增 `services.webSearch` 配置 schema（`primary` + 可选 `secondary`，每个 provider/apiKey/timeoutMs/options）。
- `app-server` 解析配置并创建 provider，通过 `ToolContributor` 注入 `WebSearchTool` 到 `core`。
- `core` 中实现 `WebSearchTool`：工具名 `WebSearch`，输入 `{query, limit, include_content}`，输出纯文本 `Title/Date/URL/Snippet/Content`。
- TUI 工具结果 chip 渲染：按结果条数显示 `N results` / `no results` / `web result`。
- 单测：provider 单元测、registry/fallback 单元测、配置解析单测、工具执行单测、TS-Rust parity 测试。

### Scope Out [C:USER]
- 不保留 Responses API hosted `web_search` 原生工具路径。
- 不保留 `web/run` namespaced extension 工具路径。
- 不保留 `OpenPage` / `FindInPage` 语义；新 `WebSearchTool` 只支持 `query` 搜索。
- 本次不实现 OAuth 刷新链路（`resolveOAuthTokenProvider` 后续补齐）。
- 不修改 LLM 协议层对通用工具 schema 的处理，只修改 web search 工具自身的 schema。
- 不引入 keyring 或环境变量注入 API key；配置中 `apiKey` 为明文字符串，与 TS 一致。

## Architecture / Design

### 总体架构 [C:USER]

```
┌─────────────────────────────────────────────────────────────┐
│  config.toml                                                │
│  [services.webSearch]                                       │
│  primary = { provider = "duckduckgo" }                      │
│  secondary = { provider = "bing", apiKey = "..." }          │
└──────────────────────┬────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  ody-config                                                 │
│  parse `services.webSearch` → `WebSearchConfig`             │
└──────────────────────┬────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  ody-app-server                                             │
│  resolveWebSearchRuntime(config) → Option<WebSearchProvider> │
│  if Some, inject WebSearchTool via ToolContributor           │
└──────────────────────┬────────────────────────────────────────┘
                       │ Arc<dyn WebSearchProvider>
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  ody-core                                                   │
│  WebSearchTool.execute() → provider.search(query, opts)     │
│  format results → plain text ToolOutput                     │
└──────────────────────┬────────────────────────────────────────┘
                       │ ToolOutput
                       ▼
┌─────────────────────────────────────────────────────────────┐
│  ody-tui                                                    │
│  chip renderer: N results / no results / web result          │
└─────────────────────────────────────────────────────────────┘
```

### Crate 边界 [C:USER]
- `ody-web-search`：trait、注册表、fallback、错误分类、配置类型、12 个 provider 实现。不依赖 `ody-core` / `ody-app-server`，只依赖 `ody-protocol` / `serde` / `reqwest` / `tracing` 等基础库。
- `ody-config`：解析 `services.webSearch` 为 `ody_web_search::WebSearchConfig`。
- `ody-app-server`：根据 `Config` 创建 `WebSearchProvider` 并注入；依赖 `ody-web-search`。
- `ody-core`：实现 `WebSearchTool` 并接入工具规划；通过 trait 调用 provider，不直接解析配置。
- `ody-tui`：纯文本 chip 渲染。

## Data Models（摘要） [C:UPSTREAM]

完整定义见 Part 1 `trait-crate.md` 与 Part 2 `registry-fallback.md`。

| 模型 | 关键字段 | 来源 |
|---|---|---|
| `WebSearchResult` | `title`, `url`, `snippet`, `date: Option<String>`, `content: Option<String>`, `raw: Option<Value>`（调试） | TS `WebSearchResult` |
| `WebSearchOptions` | `limit: Option<u32>`, `include_content: Option<bool>`, `tool_call_id: Option<String>` | TS `provider.search(query, options)` |
| `WebSearchProviderConfig` | `provider: WebSearchProviderName`, `api_key: Option<String>`, `timeout_ms: Option<u64>`, `options: Option<Map<String, Value>>` | TS `WebSearchProviderConfigSchema` |
| `WebSearchConfig` | `primary: WebSearchProviderConfig`, `secondary: Option<WebSearchProviderConfig>` | TS `WebSearchConfigSchema` |
| `WebSearchProviderName` | 12 个枚举值（Duckduckgo, Bing, ...） | TS `WebSearchProviderNameSchema` |
| `WebSearchError` | `Cancelled`, `Timeout`, `Authentication`, `RateLimit`, `Network`, `Server`, `Other`, `InvalidOptions`, `UnknownProvider` | TS `fallback.ts` 错误分类 |

- `WebSearchResult` 规范化：`title` 截断 500，`url` 截断 2048，`snippet` 截断 4000；过滤 `title`/`url` 为空；应用 `limit` slice。
- `timeout_ms` 范围 1000..=120000。
- `limit` 有效范围 1..=20；provider 可 clamp 或返回 `InvalidOptions`。

## Algorithms（摘要） [C:UPSTREAM]

完整算法与伪代码见 Part 1–5。

### A1 — 搜索执行流程（`WebSearchTool::handle_call`）
```text
1. 解析 ToolCall function arguments 为 WebSearchInput。
2. 校验 query 非空；否则返回 RespondToModel。
3. 构造 WebSearchOptions { limit, include_content, tool_call_id }。
4. 调用 provider.search(query, options).await。
5. 若 Ok(results)：
   - 空结果 → 返回 "No search results found."。
   - 非空 → 按 "Title\nDate\nURL\nSnippet\nContent" 格式化每个结果，结果间用 "---\n" 分隔。
6. 若 Err(e)：返回 FunctionCallError::Fatal(classify_search_error(e).to_string())。
```

### A2 — Fallback 决策（`FallbackWebSearchProvider::search`）
```text
1. 调用 primary.search(query, options).await。
2. 若 Ok → 返回结果。
3. 若 Err(primary_err)：
   - 若非 is_retryable_search_error(primary_err) → 返回 primary_err。
   - 若有 secondary → 返回 secondary.search(query, options).await 的结果（无论成败）。
   - 若无 secondary → 返回 primary_err。
```

### A3 — 错误分类（`classify_search_error`）
```text
输入：任意实现 Display 的错误
1. message = error.to_string().to_lowercase()
2. 若包含 "abort" → Cancelled
3. 若包含 "timed out" / "timeout" → Timeout
4. 若包含 "401" / "403" / "unauthorized" / "authentication" → Authentication
5. 若包含 "429" → RateLimit
6. 若包含 5xx 相关子串 → Server
7. 若包含 "network" / "dns" / "connection" / "fetch" → Network
8. 否则 → Other
```

### A4 — Provider 响应规范化（`normalize_results`）
```text
输入：原始 Vec<WebSearchResult>, limit: Option<u32>
1. 对每条结果：
   - 字段别名映射（name→title, link/uri→url, snippet/description/content/text→snippet）。
   - 截断 title/url/snippet 到上限。
2. 过滤 title 或 url 为空的结果。
3. 按 limit 切片（若指定）。
4. 返回规范化后的 Vec。
```

## Error Handling / Degradation（摘要） [C:UPSTREAM]

完整错误处理策略见 Part 1 `trait-crate.md`、Part 2 `registry-fallback.md`、Part 5 `tool-tui.md`。

| 场景 | 行为 | 是否 retryable |
|---|---|---|
| Primary provider 成功 | 直接返回结果 | - |
| Primary 返回非 retryable 错误（Authentication / InvalidOptions / Other） | 直接返回该错误 | 否 |
| Primary 返回 retryable 错误（Timeout / RateLimit / Network / Server） | 尝试 secondary；若 secondary 存在则返回其结果/错误 | 是 |
| 未配置 secondary 且 primary 失败 | 返回 primary 错误 | 视错误类型 |
| 输入 JSON 解析失败 | `FunctionCallError::RespondToModel`（让模型修正） | - |
| query 为空 | `FunctionCallError::RespondToModel` | - |
| provider 返回空结果 | 不是错误；返回 "No search results found." | - |
| 配置解析失败（unknown provider / timeout 越界） | 配置加载报错；`app-server` 不注入工具并记录 warning | - |
| 旧 `web_search` 配置存在 | 发出 deprecation warning，不自动迁移 | - |
| 旧 thread history 包含 `WebSearchItem` | 保留兼容降级解析层，显示为 `[legacy web search result]` | - |

### 2. 四视角 adversarial sweep

#### Security
- **API key 明文存储**：与 TS 一致，配置中 `api_key` 为 `String`。未引入 keyring 或环境变量注入。这是 TS 复刻策略的一部分，但确实增加了配置泄露风险。决策：在 `Assumptions` 中未单列，因为已在 Scope Out 中明确说明 `[C:USER]`；接受此风险，作为与 TS 一致的设计点。
- **DuckDuckGo HTML 解析**：解析不可信 HTML 可能受 XSS 影响，但解析结果仅作为文本返回给模型，不渲染为 HTML/JS。决策： provider 内部仅做文本提取，不执行任何脚本或构造 DOM。
- **Moonshot OAuth 预留**：OAuth 刷新链路本次不实现，但接口中预留 `token_provider` 为 `Option`；避免未来改动破坏 trait 签名。决策：在 `providers.md` 中明确说明 OAuth 本次不实现。

#### Test / Verification
- **测试覆盖率**：每个 provider 都有独立单元测试，但 L3 parity 只能抽样。风险：未抽样的 provider 可能与 TS 输出存在偏差。缓解：第一批必须做 parity；未抽样 provider 的响应字段映射和请求参数需与 TS 源码逐行核对，并记录到 `known-gaps.md`。
- **mock HTTP 测试稳定性**：`reqwest` 不支持 `base_url`，测试中依赖 `options.baseUrl` 覆盖。若某 provider 不支持 `baseUrl`，测试将变复杂。决策：在 `testing.md` 中要求所有 provider 支持 `options.baseUrl`（或等价测试覆盖），列为 Assumption 6。
- **旧代码删除后的回归**：旧测试将随旧代码一起删除，但新测试不覆盖旧代码被误触发。缓解：`deletion-migration.md` 验证清单要求全 workspace grep 无残留；`config` 测试验证旧 `web_search` 仅触发 warning。

#### Operations
- **配置变更动态刷新**：`ConfigContributor::on_config_changed` 会替换 `WebSearchExtensionConfig`，但如果新配置无效，旧 provider 仍保留。`config-injection.md` 中描述“如果新配置无效则移除旧配置”，但在示例代码中只做了 `insert`；需要在实现时确保无效配置时调用 `remove`。已记录为设计注意点，但不影响总体架构。
- **无 provider 时工具不出现**：依赖 `ToolContributor::tools` 在找不到 `WebSearchExtensionConfig` 时返回空 Vec。需要确保 `ThreadStore` 的 key 类型一致，否则即使插入了也查不到。已在 `config-injection.md` 中定义了 `WebSearchExtensionConfig` 类型并在同一 crate 中读写，降低风险。
- **分阶段交付**：按批次实现 provider，降低 review 负担，但第一批上线后用户可能期望所有 12 个 provider 可用。`providers.md` 已明确分三批，并作为 D1.0 关闭条件的一部分。

#### Integration
- **`ody-web-search` 不依赖 `ody-core`**：保证单向依赖。`WebSearchTool` 放在 `ext/web-search` 而不是 `core`，因此 `core` 不直接依赖 `ody-web-search`。这是正确的，但需要确认 `ody-core` 中的工具规划层是否能正确接受 `ext/web-search` 注册的工具。`app-server/src/extensions.rs` 已存在 `ody_web_search_extension::install` 调用，说明该路径已打通。
- **TUI chip 与通用工具输出**：删除 `WebSearchCell` 后，TUI 依赖通用工具调用的输出文本。`tool-tui.md` 中定义了稳定的输出格式，chip 计数基于输出文本。风险：若输出格式改变，chip 会错误。缓解：测试覆盖多种输出场景；`testing.md` 中 L1 测试包含 chip 计数。
- **配置 schema 兼容**：`services.moonshotSearch` / `moonshotFetch` 作为 `serde_json::Value` 透传，避免 TS 配置迁移时解析失败。但这也意味着 Rust 端不会校验这些字段。风险：用户可能配置错误但无提示。决策：与 TS 一致，Moonshot 相关 provider 自行读取并报错；本次不扩展校验范围。

### 3. Scope 再检查

- **Scope In**：删除旧路径、新建 `ody-web-search`、12 provider、`services.webSearch` 配置、app-server 注入、`WebSearchTool`、TUI chip、测试。全部在设计中有明确对应部分。
- **Scope Out**：Responses API hosted 原生工具、`web/run` namespaced 路径、`OpenPage`/`FindInPage`、OAuth 刷新、LLM 协议层通用工具处理改动、keyring/环境变量 API key。全部明确排除。
- **灰色地带**：旧 thread history 的 `WebSearchItem` 兼容降级层。设计在 `deletion-migration.md` 中建议保留，并在 `Assumption 2` 中列为 medium。不算 Scope Out，但属于需要谨慎处理的迁移项。
